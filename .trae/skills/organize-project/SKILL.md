# 整理工程

## 触发条件

- 当用户明确说出“整理工程”时启用本 Skill。
- 近义表达如“整体清理工程”“做一轮工程整理”“全仓扫描并清理”也可以视为启用信号。
- 如果用户只要求改某个局部文件、单个 bug、单个 README，不启用本 Skill，按普通任务处理。

## 目标

在不脱离当前实现的前提下，对工程做一轮面向可维护性的系统整理：

1. 扫描全部文件形成整体认知，不在局部上下文里盲改。
2. 清理冗余函数、重复变量、死代码、无效分支、未引用的私有辅助逻辑。
3. 识别可安全删除的冗余 Python 文件，并在确认无引用后删除。
4. 对保留代码做整洁化，保证结构、命名、注释、文档与当前实现一致。
5. 更新所有受影响函数的中文注释，以及 README、HANDOFF 等工程文档，使其反映目标态。

## 范围原则

- 默认扫描整个工作区以形成认知。
- 默认清理对象以源码、配置、README、HANDOFF、策略模板和项目规则文件为主。
- 默认不清理运行期产物、数据库文件、缓存、日志、`__pycache__`、第三方静态资源，除非用户明确要求。
- 默认不重写业务逻辑，不做与“整理工程”无关的功能新增。

## 执行顺序

### 1. 全局扫描，先理解再动手

- 先扫描目录结构、入口文件、核心模块、README、HANDOFF、配置文件、策略模板、服务层、数据层与前端入口。
- 先识别哪些文件是核心入口，哪些是模板，哪些是遗留或边缘实现。
- 形成以下认知后再改：
  - 当前真实运行链路
  - 对外 API 和前端依赖
  - 配置入口与状态持久化边界
  - 模板文件与示例文件的角色区分

### 2. 冗余代码清理

- 优先处理以下对象：
  - 未被调用的私有函数
  - 重复实现且职责重叠的辅助函数
  - 已无实际用途的变量、中间状态、过渡分支
  - 已失效的兼容代码或旧命名残留
  - 能确认无引用的冗余 Python 文件
- 删除前必须先确认引用关系。
- 对“看起来像没用，但可能通过反射、注册、字符串路径、manifest、动态导入使用”的对象，不得直接删除；必须先核实加载路径。

### 3. 代码整洁化

- 在不改变外部行为的前提下，做小步、安全的整洁化：
  - 收紧函数职责
  - 删除重复判断
  - 合并重复常量或重复辅助逻辑
  - 修正命名与注释失配
  - 保持代码风格和现有结构一致
- 不为了“好看”做大规模无收益重排。
- 不把整理任务扩展成架构重写。

### 4. 文档与注释更新

- 所有被整理过的模块，都要同步检查模块级中文注释和关键函数注释。
- README、HANDOFF、策略模板说明要和当前实现一致，不允许继续保留旧行为描述。
- 函数级注释至少写清：输入、输出、用途、边界条件。
- 文档与注释只描述目标态，不描述“本次怎么改过来的”。

## 删除安全规则

- 删除函数、类、常量、文件前，必须先检查是否存在以下引用：
  - 静态导入
  - 动态导入
  - manifest / 配置 / 字符串路径引用
  - 前端调用或模板依赖
  - 文档中仍声明为正式能力的接口
- 对以下对象默认提高谨慎级别：
  - `app/main.py`、`app/api/routes.py`、`app/settings.py`
  - `app/services/*manager.py`
  - `strategies/groups/*/manifest.json`
  - `strategies/groups/strategy_2/*`
  - 被 README / HANDOFF 说明为入口、模板或契约边界的文件
- 如果无法高置信确认删除安全性，保留代码并把疑点记录给用户，而不是冒险删。

## Python 文件清理规则

- 仅在同时满足以下条件时删除 `.py` 文件：
  1. 文件没有被静态导入或动态路径引用。
  2. 文件不属于入口、注册点、模板、迁移脚本、约定目录结构的一部分。
  3. 文件内实现没有被 manifest、配置或文档声明为正式能力。
  4. 删除后能同步修正文档、导入和目录说明。
- 若文件只是“实现质量差”而不是“完全冗余”，优先整理，不优先删除。

## README / HANDOFF 更新规则

- `README.md`：更新项目总览、运行链路、目录结构、模块职责、接口概览、依赖和状态库说明。
- `HANDOFF.md`：更新交接基线、关键链路、风险点、模板约定、文档联动边界。
- 子目录 README：只保留与当前目录真实职责相符的内容，不留空泛模板文案。
- `strategy_2` 相关 README 和注释必须继续保持“给后续 AI 写策略”的强指导口径。

## 策略模板特别规则

- `strategies/groups/strategy_2` 是高优先级模板资产。
- 整理工程时，必须检查模板说明是否仍然能指导后续 AI 正确创建新策略。
- 若实际框架约定已变化，必须同步更新：
  - `strategies/README.md`
  - `strategies/groups/README.md`
  - `strategies/groups/strategy_2/README.md`
  - `strategies/groups/strategy_2/manifest.json`
  - `strategies/groups/strategy_2/engine.py`
  - `strategies/groups/strategy_2/strategy.py`

## 输出要求

- 最终输出应说明：
  1. 删除了哪些冗余实现或文件
  2. 合并了哪些重复逻辑
  3. 更新了哪些文档与注释
  4. 哪些疑点因为删除风险高而被保留
- 如果未做测试，要明确说明未测试。

## 禁止事项

- 不在未扫描全工程前直接开始大规模删除。
- 不基于“可能没用”就删除入口、模板、注册点或策略文件。
- 不为了整理而擅自改动数据库文件、缓存文件、日志文件。
- 不把“整理工程”偷换成纯格式化。
- 不保留与目标态无关的过渡注释或历史兼容文案。
